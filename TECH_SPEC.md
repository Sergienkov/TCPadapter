# TCPadapter - структурированное ТЗ (черновик v1)

## 1. Цель
Реализовать сервис `TCPadapter`, который:
- принимает TCP-соединения от контроллеров;
- обменивается сообщениями с Kafka;
- преобразует бинарный протокол контроллеров в внутренний JSON и обратно;
- обеспечивает надежную доставку команд, контроль TTL/дубликатов и восстановление после рестарта.

## 2. Область ответственности
Сервис покрывает:
- TCP-сессию и жизненный цикл контроллера;
- буферизацию/приоритезацию команд;
- сериализацию/десериализацию бинарного протокола;
- интеграцию с Kafka (команды, телеметрия, события, ack, DLQ);
- персистентное хранение сессий и очередей;
- наблюдаемость, диагностику и обработку ошибок.

Не покрывает:
- бизнес-логику админ-панели;
- оркестрацию балансировки на уровне инфраструктуры;
- хранение мастер-данных объекта (ID/группы/расписания) как источник истины.

## 3. Функциональные требования (консолидировано)
1. До `10 000` одновременных TCP-соединений.
2. Обязательная регистрация контроллера в пределах `5` секунд после коннекта.
3. Heartbeat/жизненные сообщения и отключение при отсутствии heartbeat более `30` секунд.
4. Команды из Kafka обрабатываются через отдельный буфер на каждый контроллер.
5. Контроль уникальности команды (dedup по ключу) и актуальности (TTL).
6. Приоритезация команд минимум по 4 буферам (высокий, FW, синхронизация, запросы).
7. Exactly-once семантика на уровне адаптера: idempotent processing + подтверждения + ретраи.
8. Персистентность сессий/очередей и корректное восстановление после рестарта.
9. Поддержка нескольких версий протокола.
10. Dead-letter очередь для неисправимых ошибок.
11. Полное логирование обмена в debug-режиме (с маскированием чувствительных полей).
12. Горизонтальное масштабирование (партиционирование по контроллеру).

## 4. Формат протокола (обобщение)
Базовый кадр:
1. `start_marker` (4 байта, например `SLDN`)
2. `payload_length` (2 байта)
3. `ttl` (1 байт): `1..250` секунд, `255` бессрочно
4. `command_seq` (1 байт, 0..255)
5. `payload` (N байт)
6. `crc16_modbus` (2 байта)

Требования к обработке:
- проверка CRC до бизнес-обработки;
- строгая проверка длины кадра;
- дедуп по `(controller_id, command_seq, command_id, payload_hash)` в окне времени;
- discard команд с истекшим TTL (код причины в ack/лог).

## 5. Архитектура сервиса
## 5.1 Модули
- `tcp_gateway`: listener, accept loop, keepalive, backpressure.
- `session_manager`: состояние сессий, heartbeat, reconnect, лимиты по IP.
- `protocol_codec`: encode/decode, CRC, версионирование протокола.
- `command_scheduler`: per-controller очереди, 4 приоритета, batching по размеру буфера.
- `delivery_engine`: отправка в контроллер, retry, timeout, ожидание ack.
- `kafka_io`: consumer команд, producer телеметрии/событий/ack/DLQ.
- `state_store`: персистентные очереди, in-flight команды, offset/позиция.
- `sync_orchestrator`: команды 12..16, CRC-проверки 17/18 и пересинхронизация.
- `fw_orchestrator`: команды 19/20 + обработка статуса 4.
- `observability`: метрики, трассировка, audit/debug лог.

## 5.2 Технический принцип масштабирования
- ключевая единица маршрутизации: `controller_id` (IMEI);
- все команды/события одного контроллера должны попадать в один shard/partition;
- sticky assignment для минимизации миграции сессии.

## 6. Kafka-контракты (предложение v1)
## 6.1 Топики
1. `tcpadapter.command.in.v1` - входящие команды в контроллер.
2. `tcpadapter.telemetry.out.v1` - телеметрия/статусы от контроллера.
3. `tcpadapter.event.out.v1` - события (ID, журнал, фото-мета, системные).
4. `tcpadapter.ack.out.v1` - подтверждения/результаты доставки команд.
5. `tcpadapter.dlq.v1` - неисправимые ошибки.

## 6.2 Ключи и partitioning
- `key = controller_id`.
- количество partitions выбирается из целевого масштаба, с запасом для роста.

## 6.3 Схема `command.in` (JSON)
```json
{
  "message_id": "uuid",
  "controller_id": "imei",
  "command_id": 8,
  "priority": "high|fw|sync|query",
  "ttl_sec": 10,
  "dedup_key": "string",
  "payload": {},
  "requested_at": "2026-02-16T16:00:00Z",
  "trace_id": "string"
}
```

## 6.4 Схема `ack.out` (JSON)
```json
{
  "message_id": "uuid",
  "controller_id": "imei",
  "command_id": 8,
  "command_seq": 42,
  "status": "accepted|delivered|failed|expired|duplicate|unsupported",
  "controller_code": 0,
  "reason": "string",
  "processed_at": "2026-02-16T16:00:02Z",
  "trace_id": "string"
}
```

## 7. Модель состояний контроллера
Состояния:
1. `DISCONNECTED`
2. `CONNECTED_UNREGISTERED`
3. `REGISTERED_SYNC_PENDING`
4. `READY`
5. `FW_UPDATE`
6. `DEGRADED`
7. `BLOCKED`

Переходы:
- `TCP connect` -> `CONNECTED_UNREGISTERED`
- `register cmd=1 ok` -> `REGISTERED_SYNC_PENDING` или `READY`
- `sync needed` -> `REGISTERED_SYNC_PENDING`
- `sync done` -> `READY`
- `fw start (cmd=19)` -> `FW_UPDATE`
- `fw complete` -> `READY`
- `heartbeat timeout (>30s)` -> `DISCONNECTED`
- `auth/ip policy violation` -> `BLOCKED`

## 8. Очереди и приоритеты (согласно протоколу)
1. Высокий приоритет: `1,2,3,5,8,17,18,19,23,24`.
2. Буфер FW: `20` (активируется после `19`; буферы 3/4 стоп).
3. Буфер синхронизации: `12,13,14,15,16`.
4. Буфер запросов: `6,7,9,10,11,22,25`.

Правила:
- отправлять пакетами с учетом свободного места буфера контроллера (`status1/ack` поле размера);
- ограничение размера пакета: учитывать 2KB буфер контроллера и сетевой лимит GSM ~1.5KB на пакет;
- при конфликте процессов (FW/sync) блокировать запрещенные команды.

## 9. Минимальная карта команд для реализации v1
Сервер -> контроллер:
- `1` регистрация ответ;
- `2` reboot;
- `3` factory reset;
- `5` триггеры;
- `8` управление выходами;
- `12..16` синхронизация структур;
- `17,18` CRC-проверки;
- `19,20` обновление FW;
- `22` запрос статуса 2;
- `23` установка времени;
- `24` подтверждение;
- `25` проверка привязки.

Контроллер -> сервер:
- `1` регистрация;
- `2` статус 1;
- `3` статус 2;
- `4` статус FW;
- `5,6,16` CRC-ответы;
- `8` ID со считывателя;
- `9` выгрузка ID;
- `10` журнал событий;
- `11` ack команды;
- `12` таймштамп структуры;
- `13` запрос блока FW;
- `14` фото;
- `17` запрос информации о привязке.

## 10. Надежность и отказоустойчивость
- at-least-once из Kafka + идемпотентная обработка в адаптере;
- in-flight журнал для незавершенных команд;
- retry policy с экспоненциальной паузой;
- poison messages -> `tcpadapter.dlq.v1` с кодом причины;
- восстановление после рестарта: загрузка сессий/очередей, повторная инициализация in-flight.

## 11. Наблюдаемость
Метрики:
- активные TCP-сессии;
- handshake success/fail;
- p50/p95/p99 latency по командам;
- доля expired/duplicate/failed;
- размер очередей по приоритетам;
- retry count, dlq rate.

Логи:
- correlation через `trace_id`, `controller_id`, `command_seq`;
- отдельный debug-канал протокольного обмена;
- маскирование IMEI/телефонов/SMS в production-логах.

## 12. Открытые вопросы к согласованию
1. Финальная схема авторизации/шифрования IMEI и канала.
2. Точный таймаут принудительной синхронизации времени (`N` секунд).
3. Канонический список запретов команд во время sync/FW (в CSV есть частично).
4. Поведение при конфликте TTL и "бессрочных" команд (`ttl=255`).
5. Единый источник истины для таймштампов структур и версия их инкремента.
6. Границы ответственности между TCPadapter и upstream API по валидации payload.

## 13. Этапы реализации
1. Каркас сервиса: TCP + Kafka + storage + observability.
2. Реализация codec и базовых команд (`1,2,11,22,23,24`).
3. Scheduler с приоритетами и TTL/dedup.
4. Синхронизация структур (`12..18`).
5. FW pipeline (`19,20`, статус `4`).
6. Фото/журнал/ID выгрузки (`8,9,10,14`).
7. Нагрузочные и отказоустойчивые тесты, hardening.

## 14. Критерии приемки (MVP)
1. Удерживает 10k подключений на целевой инфраструктуре.
2. Корректно обрабатывает регистрацию, heartbeat и reconnect.
3. Доставляет команды с учетом TTL/dedup/priority и фиксирует ack.
4. Восстанавливает состояние очередей после рестарта без потери консистентности.
5. Публикует в Kafka telemetry/event/ack/dlq по согласованным контрактам.
6. Поддерживает минимум одну полную синхронизацию структур и один сценарий FW обновления.
